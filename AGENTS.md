# Agent Global Instructions

本文件定义本仓库内通用的 Agent 全局系统指令。除非用户明确要求例外处理，否则所有开发任务都必须遵守以下规则。

## 1. 开发前先同步最新代码

在开始任何开发、修复、重构、联调、文档修改之前，必须先执行以下步骤：

1. 检查当前工作区状态，确认是否存在未提交修改。
2. 从远端拉取当前分支的最新代码。
3. 只有在完成最新代码同步后，才能开始正式开发。

执行要求：

- 不允许跳过拉取最新代码这一步直接开发。
- 不允许在未说明风险的情况下覆盖本地改动。
- 如果因为本地未提交修改、远端更新、rebase/merge 中断、权限问题或其他原因导致无法顺利拉取，必须立即停止开发并向用户说明情况。

## 2. 出现冲突时必须先沟通

如果拉取最新代码后出现 merge conflict、rebase conflict，或者任何需要人工判断的代码冲突，必须遵守以下规则：

1. 先向用户明确说明冲突原因。
2. 列出受影响的文件和冲突背景。
3. 说明当前有哪些可选处理方案。
4. 询问用户希望采用哪种解决方式。

执行要求：

- 不允许在未经用户确认的情况下自行选择冲突解决策略。
- 不允许静默保留本地版本、静默采用远端版本，或静默删除冲突内容。
- 必须等用户给出明确指示后，再继续下一步。

## 3. 用户指导后先解决冲突，再继续开发

在用户对冲突给出处理意见后，必须先完成冲突解决，再继续进行功能开发。

执行顺序必须是：

1. 根据用户要求解决冲突。
2. 确认冲突标记已清除，代码处于可继续开发状态。
3. 再开始后续开发、修复、测试或文档整理。

执行要求：

- 不允许带着未解决冲突继续开发。
- 如果用户给出的方案与当前代码状态不一致，必须再次向用户说明并确认。
- 如果冲突解决过程中发现新的风险或新的冲突，也必须继续向用户同步。

## 4. 开发完成后必须做自检查

所有开发完成后，必须严格对照 [docs/collaboration-guide.md](docs/collaboration-guide.md) 进行自检查。

至少检查以下内容：

- 前端路由、导航、入口文案是否一致。
- 新接口是否已挂载到 `server/src/index.ts`。
- `src/services/api.js` 是否同步更新。
- 新增环境变量是否更新到对应的 `*.env.example`。
- 文档是否需要同步更新。
- 是否执行了相关的 `build`、`lint` 或其他必要验证。

执行要求：

- 自检查未完成，不得进入提交阶段。
- 如果检查发现问题，必须先修复问题，再重新检查。

## 5. 自检查通过后提交并推送

当开发、自检查、必要验证全部完成且没有遗留问题后，必须执行以下步骤：

1. 写清晰、准确的 commit message。
2. 提交本次修改。
3. 将当前分支推送到远端。

执行要求：

- commit message 不得含糊不清，不得只写 `update`、`fix`、`change` 一类低信息内容。
- 推送失败时，必须向用户说明失败原因，不得假装已完成。
- 如果存在用户未确认的冲突、验证失败、关键风险未关闭，则不得提交和推送。

## 6. 沟通原则

- 对 git 冲突、拉取失败、提交失败、推送失败等问题，必须明确说明原因和下一步建议。
- 对需要用户决策的事项，必须先问清楚再执行。
- 对已经完成的检查、提交和推送，要明确反馈结果，不得省略关键状态。
